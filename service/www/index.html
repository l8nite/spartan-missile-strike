Introduction

This document contains the design of the Spartan Missile Strike Service (SMSS or service for short) a collection of APIs, data schemas, and storage systems which are custom-built to support the Spartan Missile Strike project.

The SMSS is written as a traditional client-server model. It is built as a set of HTTP APIs running on a networked appliance and talking to a storage system. The block level diagram is simplistic and included here only for completeness

 

The SMSS is intended to provide the following services to the client:

-          User authentication

-          Creating new games

-          Keeping track of game state

-          Calculating missile trajectories

-          Logging

Later sections in this document will describe in detail the APIs and data schemas required to enable and utilize the above scenarios. This document also covers the technologies used, information about the organization of the source code, and also operational information on deployment, versioning, and so forth.

Standing on the shoulders of giants

This section lists some of the more prominent technologies / libraries used in the SMSS.

Node.js
http://nodejs.org/

Node.js is a platform built on Chrome's JavaScript runtime for easily building fast, scalable network applications. Node.js uses an event-driven, non-blocking I/O model that makes it lightweight and efficient, perfect for data-intensive real-time applications that run across distributed devices.

Node.js provides an engine for flexible socket-based application development upon which the rest of the SMSS is built.

 

Restify
http://mcavage.github.com/node-restify/

restify is a node.js module built specifically to enable you to build correct REST web services. It borrows heavily from express (intentionally) as that is more or less the de facto API for writing web applications on top of node.js

Restify will be the core of our service API, helping us write a scalable, observable, web service that follows the best-practices of writing REST services.

 

Redis
http://redis.io/

Redis is an open source, advanced key-value store. It is often referred to as a data structure server since keys can contain strings, hashes, lists, sets and sorted sets.

Redis is the database used by all components of the SMSS for storing game data, user data, active sessions, statistics, etc. It is an in-memory database with options for persisting data using an append-only journal and/or interval snapshots.

 

node_redis, hiredis, and connect-redis
https://github.com/antirez/hiredis
https://github.com/mranney/node_redis

This is a complete Redis client for node.js. It supports all Redis commands, including many recently added commands like EVAL from experimental Redis server branches.

These are the client-library bindings for talking to Redis servers from Node.js applications. hiredis is a native-C redis client implementation which node_redis can take advantage of to improve performance. Finally, connect-redis provides a session store backed by a Redis server, which we use in our express.js application to handle session authentication and authorization.

Identifiers

Identifiers in the service are simply UUID (v4).

More information on UUID v4 (Wikipedia).

We'll likely be using the node-uuid module (https://github.com/broofa/node-uuid) to generate the identifiers. There is also an extremely compact example.

In the database, they are stored with a prefix (i.e., "user: 7cb07358-d486-46c4-a8a1-fb7e7c943604"); however, when used within the API, only the UUID portion is used.

Session Management

Sessions are the primary authorization mechanism for all APIs in the SMSS. In order for an SMSS API to authenticate and authorize your request, you must send a valid session key along with every request as the value of a header named 'SMSS-Session-ID', excepting of course the authentication API itself.

In the event that the SMSS is unable to authorize the session ID, an "HTTP 401 Unauthorized" response code will be returned as described in the API introduction.

To log in and receive a session ID, the client must send the user's identifier and authorization token via http POST to the /sessions resource.

 

/sessions

The /sessions API is used to authenticate a user with the service and receive a session identifier which can be used in all subsequent API calls to authorize the client.

Logging In

When a user authenticates with facebook using the local client SDK (for example, on iOS it can be either via Mobile Safari/OAUTH or the iOS SSO SDK), they receive an access token from Facebook which can then be redeemed server-side to get an access token that allows us to query their social graph.

To authenticate the client using a user's facebook credentials, first obtain a Facebook authorization code by invoking the oauth or native sign-in flow and then send the code via POST to the /sessions endpoint.

Currently, Facebook authentication is the only supported authentication method.

Example Request

POST /sessions HTTP/1.1
Content-Type: application/json; charset=utf-8


{ facebook: abc1234ef... }

 

Inputs

facebook: The authorization code obtained from the Facebook sign-on SDK.

Example Response

HTTP/1.1 200 OK

 

{ session: { id: "a74f733635a19aefb1f7..." },

user: { id: "c5172a...",newUser: true } }

Outputs

session.id: The new session identifier (to be used in the SMSS-Session-ID header for all subsequent API interactions)

user.id: The user identifier for the user account associated with this Facebook user

user.newUser: A Boolean flag indicating whether or not we just created a new user account for this facebook user. If true, you may want to prompt the user to pick a username (we set a sensible default)

Logging out

To log out of a session, simply send a DELETE request to /sessions/:id, where :id is the session identifier.

Example Request

DELETE /sessions/a74f733635a19aefb1f7 HTTP/1.1

Data Design

Redis is more than just a key-value data store, and so we should exploit its ability to store values as lists, sets, ordered sets, hashes, and more in order to fully optimize the database performance of the service.

This section of the document serves two purposes. First, it describes the JSON structure of objects returned by various API calls in the service. Second, it describes how those objects are mapped to and from the Redis data store along with example queries.

Finally, the data models and structures described here are not formalized schemas, but should provide enough information (along with their corresponding examples) for the SMSS client developers to properly implement calling the SMSS API.

Identifiers

Unique identifiers within the database system are most often UUIDs, often prefixed with a literal string representing the identifier's purpose and a colon.

For example, user identifiers are stored as "user:<UUID>"
(i.e., "user: 7cb07358-d486-46c4-a8a1-fb7e7c943604")

Sessions

A session is the basic authentication mechanism within SMSS. Once a client has an established session token, all subsequent requests to the service will validate that the clients session token matches a stored session token associated with the given userid.

Session identifiers are SHA256 values generated by encrypting a random value and the current time along with a secret key and the users identifier.

Sessions will be stored as Redis values, keyed off the session identifier itself with a prefix of session:

An important consideration when writing session-based authentication mechanisms is ensuring that the session token is not used indefinitely. We will expire the session key after it has been in use for a period of 720 hours (30 days) and require re-authentication if the client attempts to use an invalid session.

Determining if a session exists

The most common query against the session store is does this session exist? For that purpose, we can use the Redis exists command to determine if weve stored a key for the session identifier.

redis 127.0.0.1:6379> exists session:a74f733635a19aefb1f73e5947cef59cd7440c6952ef0f03d09d974274cbd6df

(integer) 1

Getting information about an existing session

A session is stored as a value in Redis underneath the key associated with the session identifier. You can query the user id of the session using the get command

redis 127.0.0.1:6379> get session:a74f733635a19aefb1f73e5947cef59cd7440c6952ef0f03d09d974274cbd6df

"user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e"

Creating a new session

Creating a new session is as simple as adding a new value using set

redis 127.0.0.1:6379> set session:a74f733635a19aefb1f73e5947cef59cd7440c6952ef0f03d09d974274cbd6df userid 0c24a4c1-7dc8-486b-a6c9-e99763fc748e

OK

Enforcing session expiration

Redis provides a facility for automatically expiring keys, so we will take advantage of it to automatically expire sessions older than 2592000 seconds (30 days).

redis 127.0.0.1:6379> expire a session:74f733635a19aefb1f73e5947cef59cd7440c6952ef0f03d09d974274cbd6df 2592000

OK

Deleting an existing session

An existing session can be deleted by using the del operation

redis 127.0.0.1:6379> del session:a74f733635a19aefb1f73e5947cef59cd7440c6952ef0f03d09d974274cbd6df

OK

Users

There are a number of details we need to store about each user, but the most critical is their identifier. The user identifier is a UUID generated when we initially create their account.

Users are stored as stringified JSON hashes keyed off the UUID with a prefix of user:.

"user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e" = {

username: "shaun.guth",

created: "1350036847",

session: "session:a74f7... ",

facebook_token: "116122545078207|EyWJJYqrdgQgV1bfueck320z7MM.",

facebook_id: "1b52a4c1-7dc8-486b-a6c9-e99763fc748e",

};

 

We also maintain a mapping of facebook identifiers to user identifier, so that we can quickly determine if a given facebook user has an account in our system. These are stored as key/value pairs in Redis, with the key being the facebook ID prefixed with "facebook:"

"facebook: 7e52a4c1-7dc8... " = "0c24a4c1-7dc8-486b-a6c9-e99763fc748e"

 

Games

A game is considered the interaction between two users.

All games have a unique identifier (UUID), and game information is stored in Redis as a stringified JSON value keyed off the game identifier with the prefix "game:"

"game:2dc0814f-c736-4c83-88aa-8b5749c15aa9": {

status: "active" (or "completed"),

created: timestamp,

creator: "user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e",

opponent: "user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e",

current: "user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e",

"user:0c24a4c1-7dc8-486b-a6c9-e99763fc748e": {

base: {

latitude: "58.586581",

longitude: "-72.432861"

},

shots: [

{

origin: {

latitude: "58.586581",

longitude: "-72.432861"

},

altitude: "",

heading: "12.3456",

power: "80"

}

]

},

"user:ac3c6272-12d9-4d58-986f-a291f994f7e6": { ... }

 

The list of games a user is participating in is stored in Redis as a zset, sorted by recency, keyed off the user's ID with a prefix of "games:"

"games:0c24a4c1-7dc8-486b-a6c9-e99763fc748e" = [

// zset of game identifiers, scored by last update timestamp
];

 

Games which have been inactive for 7 days are automatically forfeited. The service will do this check whenever a user requests a list of games which includes the to-be-forfeited game. The game record will be updated as forfeited before the data entry is returned.

 

Operations

For development and initial beta testing of the Spartan Missile Strike Service, were going to host the service on a single VPS running Ubuntu Linux. The service, database, games website, DNS, etc., running under the missileapp.com domain will be hosted on this single box. The VPS is run by hostigation.com and located in Los Angeles, CA.

For production, these components do not necessarily need to be hosted on the same machine (or even geographically related). Node.JS will run in most environments, and weve successfully prototyped a running Node instance in the Azure cloud infrastructure. Other potential hosting environments include AWS or Heroku.

Service instance hostnames will be incrementally named serviceNN.missileapp.com (i.e., service01.missileapp.com).

The service will be hosted at https://api.missileapp.com/ and load-balanced as necessary via a round-robin DNS entry to multiple hosts. The development service will run on port 8443, but the production service will run on 443.

The database instances will be distributed via a technique known as sharding, which will ensure that requests for data for a particular user are always sent to the correct instance. The redis server will listen on port 6379 (the default Redis port).

Database instance hostnames will be incrementally named dataNN.missileapp.com, and will be accessible via the same domain name.
